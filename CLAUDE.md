# AUTO-content-system — 最強の有料note構築マシーン

## プロジェクト概要

有料noteコンテンツを企画→設計→執筆→販売まで一気通貫で構築するシステム。
マーケティング・コピーライティングの知識を統合した6フェーズパイプライン。

## ディレクトリ構成

```
AUTO-content-system/
├── CLAUDE.md              ← このファイル（プロジェクト設定）
├── README.md              ← プロジェクト説明
├── post_to_note.py        ← note.com 下書き投稿（Selenium）
├── count_chars.py         ← 文字数カウント（wc -m はバイト数を返すため使用禁止）
├── start_webapp.bat       ← Web UI 起動（ダブルクリック）
├── webapp/                ← ローカルWebアプリ（claude CLI をヘッドレス起動）

│   └── static/index.html  ← チャットUI
├── knowledge/             ← あなただけの秘伝ノウハウ（裏メニューで管理・執筆時に注入）
├── prompts/               ← 各フェーズのプロンプト定義書
│   ├── 00_master_orchestrator.md  ← 全体統括
│   ├── 00a_self_analysis.md       ← Phase 0: 自己分析・棚卸し

│   └── 05_sales_letter.md         ← Phase 5: セールスレター
└── output/                ← 生成物の保存先
```

## スキルの使い方

各スキルは対応するフェーズのプロンプトに従って実行する。

- **メニュー1（自己分析）** → Phase 0: モード選択（簡易/詳細）→ 簡易=テンプレ+2〜3問（約1分）/ 詳細=会話形式で13問（約5～10分）（`prompts/00a_self_analysis.md` に従う）
- **「殴り書きモード」**（Web UIの「🖊️ 殴り書きで始める」）→ ヒアリングせず、貼り付けられた自由文をAIが解析して Phase 0 を自動構築（`prompts/00a_self_analysis.md` PART C）。その後は通常どおり実行モード選択ゲートへ進む
- `/research` → Phase 1のプロンプトに従い、Web検索で競合リサーチを実行（クイック/詳細の2モード）
- `/persona` → Phase 0+2を組み合わせ、ターゲットペルソナを設計
- `/outline` → Phase 3のプロンプトに従い、章構成・カリキュラムを設計
- `/write` → Phase 4のプロンプトに従い、たこすけ式構成法で本文を執筆
- `/proofread` → 全フェーズの品質基準に照らして校正チェック
- `/title` → Phase 3 PART Dのプロンプトに従いタイトルを生成
- 「noteに投稿して」→ `post_to_note.py` を**AI自身がツール（Bash）で実行**して note.com に下書き投稿（Selenium）。コマンドを表示するだけで終わらせない
  - 実行: `python post_to_note.py --dir {ワーキングフォルダ} --title "{タイトル}" --profile "{accounts.jsonのcurrent}" --auto-save`
  - 05_sales_letter.md（無料部分）+ 04_paid_content.md（有料部分）を結合して投稿
  - タイトルは 03_product_design.md から自動取得（明示推奨）。`--auto-save` で確認なし下書き保存

## ナレッジ機能（あなただけの秘伝ノウハウ）

`knowledge/` フォルダに保存したユーザー独自のノウハウを、執筆時に「独自要素」として注入する仕組み。
コンテンツ融合法（競合3商品＋独自要素）の「独自要素」をストック化したもの。

### ファイル規約
- `knowledge/` 内にジャンル別の `.md` ファイルで保存（例: `恋愛-婚活_ノウハウ.md`）
- 各ファイル先頭に frontmatter を付ける:
  ```yaml
  ---
  title: タイトル
  genre: ジャンル名          # Phase 0 のジャンルと照合する
  tags: [タグ, タグ]
  type: ノウハウ
  priority: high | medium | low   # high は競合知識より優先して必ず使う
  ---
  ```
- `knowledge/index.md` は目次。ファイルを追加/削除したら表を更新する。

### 裏メニュー（ナレッジ管理）
ユーザーが「ナレッジ」「ナレッジ管理」と入力するか、Web UI右下の📚ボタンを押したら、以下のメニューを表示する:
```
============================================================
  📚 ナレッジ管理（あなただけの秘伝ノウハウ）
============================================================
  1. 一覧を見る        （登録済みノウハウをジャンル別に表示）
  2. ノウハウを追加     （テキストを貼り付け → AIが整形して保存）
  3. ノウハウを編集     （既存ファイルに追記・修正）
  4. ノウハウを削除
  5. 戻る
============================================================
```
- **1 一覧** → `knowledge/index.md` と各ファイルの frontmatter を読んで一覧表示
- **2 追加** → ユーザーが貼り付けたテキストを受け取り、AIがジャンルを判定 → 該当する既存ファイルに追記、無ければ新規作成。frontmatter を自動付与し、`index.md` の表も更新する
- **3 編集** → ファイルを選ばせて該当箇所を修正
- **4 削除** → ファイルを選ばせて削除し、`index.md` からも除外

### 執筆時の自動参照（ハイブリッド方式）
1. **実行モード選択・リサーチモード選択が終わった後、Phase 1 を始める前**に、選択ジャンルと `genre` が一致するナレッジを自動検出し、ユーザーに提案する（フルオート/ステップ確認どちらのモードでも、この提案だけは表示してよい。これは制作前の準備選択であり、Phase 1〜5 の確認ではないため）:
   ```
   📚 このジャンルのノウハウが見つかりました。本文に反映しますか？
     ・恋愛-婚活_ノウハウ.md（★high）
     ・ライティングの型.md（medium）
   → [このまま使う / 追加・除外を指定 / 使わない]
   ```
   - 一致が無ければこのステップは黙ってスキップ（提案を出さない）
   - ユーザーが追加/除外を指定したら反映する
2. 選択されたナレッジを Phase 2・3・4 の実行時にプロンプトへ注入する
3. **Phase 4 本文執筆**では、競合3商品の良い要素 ＋【ユーザーのノウハウ】を融合する。`priority: high` のノウハウは必ず本文に織り込み、AIの一般論より優先する

### 出力時の鉄則（痕跡を残さない）
- ナレッジの「中身（ノウハウ）」は本文に反映してよい
- ただし **ファイル名・frontmatter・タグ・「ナレッジを参照した」という旨は note 等の最終出力に一切出さない**。コンセプトのワーディングに自然に溶け込ませる
- 「priority」「genre」などのメタ用語も本文に登場させない

## セールスレターの型（複数テンプレート）

`sales_templates/` フォルダに、Phase 5 で使う「型」を複数保存する。デフォルト型は常に残す。

### ファイル規約
- `sales_templates/` 内に型ごとの `.md`。先頭に frontmatter:
  ```yaml
  ---
  name: 型名
  summary: 一行説明
  when: 向いている商品
  builtin: true | false   # true は標準同梱（デフォルト候補）
  ---
  ```
- 本文に「セクション構成・狙い・ルール」を書く。`sales_templates/index.md` は目次。
- 同梱: 王道17セクション感情型（builtin・デフォルト）/ PASONA型 / 実績ドリブン型 / ストーリーテリング型 / スキャナブル箇条書き型

### 型の追加（お手本レターの逆解析）
- **「レターの型を追加」** と入力 or 📄→「型を追加」ボタン → 続いて**お手本にしたいセールスレター全文**（noteの無料部分など）が貼り付けられる
- 受け取ったレターに対して **`prompts/reverse_engineer_letter.md` の手順を実行**し、「誰でも再現できる型」を生成する
- 生成した型（frontmatter付きMarkdown）を `sales_templates/{name}.md`（`builtin: false`）に新規保存し、`index.md` の表を更新する
- 完了したら「型『◯◯』を追加しました。右上のプルダウンから選べます」と短く報告

### 型の切り替えと Phase 5 での使用
- Web UI 右上の「レターの型」プルダウンでいつでも切替（選択は `sales_templates/.current` に保存される）
- **Phase 5 開始時、`sales_templates/.current` を読んで現在選択中の型を使う**。未設定・空なら `builtin: true` のデフォルト型を使用
- ステップ確認モードで未選択なら型一覧を提示して選ばせてもよい
- 選択された型ファイルの構成に従ってレターを書く。デフォルト型なら `prompts/05_sales_letter.md` の17セクション構成を使う

### 型の管理UI
📄ボタン →「確認・編集」でモーダル一覧・プレビュー・編集・保存（`sales_templates/*.md` のみ編集可）

## 品質チェック機能（ワンポチ）

Web UI のボタン、またはコマンド入力で起動する。対象は原則「最新のワーキングフォルダ」（output が複数あれば最大番号）の生成物。

### 多視点レビュー（コマンド「多視点レビュー」/ 🔍ボタン）
最新フォルダの `04_paid_content.md` と `05_sales_letter.md` を、3つの視点で**並行評価（サブエージェント可）**し、採点と改善案を出す。
1. **読者視点**（100点）: 刺さるか／続きが気になるか／わかりやすいか／自分ごと化できるか → 点数＋具体的改善3つ
2. **セールス視点**（100点）: 買いたくなるか／3つの刷り込み・ザイオンス（キーメッセージ7回）・17セクション感情フロー・CTAが機能しているか → 点数＋改善3つ
3. **校正視点**: 誤字脱字・表記ゆれ・論理矛盾・スマホ可読性（1段落4行）・**表が混入していないか**・見出し階層 → 指摘リスト
最後に **総合スコア** と「直すと一番効く改善 Top3」を提示。結果は `{ワーキングフォルダ}/review.md` に保存する。

### 離脱診断（コマンド「離脱診断」/ 📉ボタン）
最新フォルダの `05_sales_letter.md`（無料部分）を対象に、読者が読むのをやめそうな **離脱ポイント**を特定する。
- 各離脱ポイントについて「なぜ離脱しそうか」を説明
- 各箇所に「続きが気になるクリフハンガー（引き）」の改善案を提示
- **無料→有料の境界の引き**（有料エリア直前で最も読みたくさせているか）を診断し、強化案を出す
- 結果は `{ワーキングフォルダ}/dropout_check.md` に保存する

## フォルダ自動採番

コンテンツ制作を開始する際（Phase 0 のヒアリング開始時）、以下のルールでワーキングフォルダを決定する。

### 【最優先】ワーキングフォルダが指定されている場合
メッセージ冒頭に **「【このセッションのワーキングフォルダは outputN/ です…】」** という指定があれば、**そのフォルダをそのまま使い、新しいフォルダは作らない**（Web UIの「＋ 新規」で事前に作成済み）。下記の自動採番は行わない。

### 自動採番ルール（指定が無いときのみ）
1. `output/` フォルダが**存在しない** → `output/` を作成してそこに保存
2. `output/` フォルダが**既に存在する** → 次の枝番フォルダを自動作成
   - 既存フォルダを走査: `output2/`, `output3/`, ... を確認
   - 最大番号+1 のフォルダを作成（例: `output2/` まであれば `output3/` を作成）
3. メニュー「11. 追加コンテンツ」を選択した場合も同じ自動採番ルールで新規フォルダを作成

### フォルダ命名規則
```
output/       ← 初回コンテンツ
output2/      ← 2回目
output3/      ← 3回目
output4/      ← 4回目
...
```

### 動作
- フォルダ作成時にユーザーへ「📁 ワーキングフォルダ: output3/ を作成しました」と通知する
- 各フェーズの出力は現在のワーキングフォルダに保存する
- 追加コンテンツも同じ6フェーズパイプラインで制作する

## noteアカウント管理

アカウント情報は `accounts.json` で管理する。

### accounts.json の構造
```json
{
  "current": "Profile 1",
  "accounts": [
    { "profile": "Profile 1", "name": "メインアカウント" }
  ]
}
```

### アカウント切り替えメニュー（メニュー10選択時）
```
============================================================
  noteアカウント管理
============================================================

  登録済みアカウント:
  ● 1. メインアカウント（現在使用中）
    2. サブアカウント

------------------------------------------------------------
  A. アカウントを追加（新しいnoteアカウントでログイン）
  B. 戻る（メインメニュー）
============================================================
```

- **番号を選択** → `accounts.json` の `current` を切り替え、投稿時にそのプロファイルを使用
- **Aを選択** → アカウント名を入力後、`python setup_note.py --profile "Profile N"` を実行。ログイン完了後に `accounts.json` に追加
- **Bを選択** → メインメニューに戻る

### 投稿時のアカウント連携
`post_to_note.py` 実行時は `accounts.json` の `current` プロファイルを自動的に `--profile` に渡す。

## 実行モード

**【重要：絶対スキップ禁止のゲート】**

Phase 0 のヒアリングが完了したら、**必ず**実行モード選択メニューを表示し、ユーザーの回答を待つこと。

**禁止事項:**
- ユーザーが「1」または「2」を明示的に選択するまで、Phase 1以降に進んではならない
- 実行モード選択を省略してはならない
- ヒアリング完了の流れで勝手にPhase 1を開始してはならない
- 「フルオートで進めますね」等、AIが勝手にモードを決定してはならない

| モード | 説明 |
|--------|------|
| 🚀 フルオートモード | ヒアリング後、確認なしでPhase 1〜5 → note投稿まで一気に実行 |
| 📋 ステップ確認モード | 各フェーズ完了時に確認を取り、修正指示を受けてから次へ進む |

- フルオートモード: 全フェーズを自動実行し、完了後にまとめて結果を報告する
  - **⛔ フルオート中の絶対禁止事項:**
  - Phase 1〜5の途中でユーザーに確認を求めてはならない
  - 「次に進みますか？」「よろしいですか？」等の確認質問を一切してはならない

  - note投稿完了まで一切の中断・確認なしで実行し続けること
  - **⛔ Phase 5 完了後、note投稿コマンドを「表示」して止まってはならない。AI自身がツール（Bash）で `python post_to_note.py --dir {ワーキングフォルダ} --title "{03のメインタイトル}" --profile "{accounts.jsonのcurrent}" --auto-save` を実行し、下書き投稿まで完了させること**
  - エラーが発生した場合のみ停止してユーザーに報告する

## 実行ルール

1. 各フェーズは必ず前フェーズの出力を参照してから実行する
2. コンセプト（Phase 2）が全フェーズの基準になる。矛盾が生じたらPhase 2を優先
3. 出力は現在のワーキングフォルダ（`output/` または `output2/`, `output3/`...）に保存する
4. 品質基準は `00_master_orchestrator.md` の品質基準表に従う
5. スマホ最適化: 1段落4行以内、適切な改行・装飾
6. **note投稿対象の最終アウトプット（04_paid_content.md / 05_sales_letter.md）ではMarkdownの表（`|---|`形式）を使用禁止**。noteのエディタは表に対応していないため、比較・一覧は箇条書きや「項目: 説明」形式で表現する（Phase 0〜3の内部設計書では表を使ってよい）
7. **文字数の実測は必ず `python count_chars.py <ファイル>` で行う。`wc -m` は使用禁止**（Windows環境では日本語をバイト数=約3倍でカウントしてしまうため）。基準判定・完了レポートには「空白・改行除く」の数値を使う
8. **【必須ゲート】Phase 0 完了後、実行モード選択メニューを表示してユーザーの選択を待つ。選択されるまでPhase 1に進んではならない**
9. **【必須ゲート】各STEPの質問はユーザーの回答を待ってから次に進む。先回りして複数ステップを同時に実行しない**
10. **ナレッジ参照**: モード選択完了後・Phase 1開始前に、`knowledge/` から選択ジャンルに一致するノウハウを自動検出して提案する（一致が無ければ黙ってスキップ）。選択されたノウハウは Phase 2〜4 に注入し、`priority: high` は本文で必ず使う。ただし**ナレッジの存在・ファイル名・メタ情報は最終出力（note）に一切出さない**
